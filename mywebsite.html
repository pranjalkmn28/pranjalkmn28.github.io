Notes on the work

The fastest API is the one you don't have to make.

Most of the latency wins I'm proud of are from work the request never did. A read-through cache, a precomputed aggregate, an N+1 unwound into a single join. Code that doesn't run is code that doesn't break.

Slow queries don't get faster on their own.

Half of backend performance is paying attention to what the database is doing — query plans, locks, indexes you thought you had but didn't. The boring part is where the wins are.

Good incident response fixes the alert. Better incident response fixes the alert class.

Production fires teach you things staging never will. The job isn't to make a page go away — it's to ask why this kind of page ever fired.

Selected work

Careers360Principal Software Engineer
2025 — present

When I joined, the top ten revenue-critical services were running at a p95 around 1.2 seconds. We brought that down to roughly 480 ms — a 60% cut across the board — mostly through three moves: a Redis read-through layer for hot paths, surgical elimination of N+1 queries the ORM had been quietly making for years, and migrating synchronous third-party calls to Celery so we stopped blocking the request thread on someone else's outage.

I also run the Rapid Action Team. Mean time to resolution went from about fourteen hours to under four. Some of that's better tooling; most of it is better postmortems.

ZopperSr. Software Engineer

Bancassurance is the unsexy word for banks selling insurance through their own channels. We built the platform — assisted and self-serve — that powers that for several Indian banking partners. End-to-end digital journeys, tens of thousands of policy issuances a month.

The interesting parts were the joints: connecting a partner bank's KYC stack to an insurer's underwriting engine to a regulator's compliance requirements, while keeping the user experience down to a couple of screens.
